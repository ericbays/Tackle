# 14 — Database Schema Design

## 1. Purpose

This document defines the complete database schema for the Tackle phishing simulation platform. The schema is the single source of truth for all persistent state in the system — users, authentication, campaigns, targets, infrastructure, credentials, tracking events, logs, reports, and AI readiness structures. Every table, column, type, constraint, index, and relationship is specified here. The schema is designed for PostgreSQL with a schema-first approach, enforced referential integrity, and encryption at rest for all sensitive fields.

---

## 2. Design Principles

**REQ-DB-001** — The database engine SHALL be PostgreSQL (version 14 or later).

*Acceptance criteria:* All DDL statements in the schema are valid PostgreSQL syntax. No proprietary extensions from other database engines are used.

**REQ-DB-002** — All tables SHALL use UUID primary keys generated by the application (UUIDv4). Sequential integer IDs SHALL NOT be used for any table to prevent resource enumeration attacks.

*Acceptance criteria:* Every table's `id` column is of type `UUID` with a `DEFAULT gen_random_uuid()`. No `SERIAL` or `BIGSERIAL` columns exist in the schema.

**REQ-DB-003** — All tables SHALL include `created_at TIMESTAMPTZ NOT NULL DEFAULT now()` and `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()` columns, unless the table is an immutable junction table or an append-only log table (which requires only `created_at`).

*Acceptance criteria:* Every table definition includes the required timestamp columns. A database trigger or application-level hook sets `updated_at` on every UPDATE operation for tables that have it.

**REQ-DB-004** — Referential integrity SHALL be enforced via foreign key constraints on all relationships. Orphaned records SHALL NOT be possible through normal application operations.

*Acceptance criteria:* Every foreign key column has an explicit `REFERENCES` clause. ON DELETE behavior is specified for every foreign key (CASCADE, SET NULL, or RESTRICT as appropriate for the relationship).

**REQ-DB-005** — Soft deletes SHALL be implemented on campaigns, targets, target groups, domains, email templates, landing pages, and SMTP configurations. Soft-deleted records SHALL retain all data and remain queryable for historical reporting. Hard deletes SHALL NOT be permitted on these tables through the application layer.

*Acceptance criteria:* Tables requiring soft delete include a `deleted_at TIMESTAMPTZ` column (NULL when active). Application queries filter on `deleted_at IS NULL` by default. No DELETE statements are issued against soft-delete tables in normal operation.

**REQ-DB-006** — JSONB columns SHALL be used for flexible, extensible data where the schema of the content varies per record or is expected to evolve without migrations (e.g., configuration blobs, custom fields, form data, metadata).

*Acceptance criteria:* JSONB columns are used where specified in the table definitions below. JSONB columns have documented example payloads in code comments or developer documentation.

**REQ-DB-007** — All sensitive fields (API credentials, SMTP passwords, captured form data, provider configurations containing secrets) SHALL be encrypted at rest using application-level AES-256-GCM encryption before storage. The encryption key SHALL be provided via environment variable and SHALL NOT be stored in the database.

*Acceptance criteria:* Encrypted columns are stored as `BYTEA` or `TEXT` (base64-encoded ciphertext). Decryption occurs exclusively in the application layer. The encryption key never appears in the database, logs, or migration files.

**REQ-DB-008** — All enum-like status columns SHALL be implemented as PostgreSQL `TEXT` columns with `CHECK` constraints enumerating the valid values. Native PostgreSQL `ENUM` types SHALL NOT be used (they are difficult to modify in migrations).

*Acceptance criteria:* Every status or type column has an accompanying `CHECK` constraint. No `CREATE TYPE ... AS ENUM` statements exist in the schema.

## 3. Schema Overview

The schema is organized into the following logical groups:

| Group | Tables | Purpose |
|-------|--------|---------|
| **Authentication & Authorization** | `users`, `roles`, `permissions`, `role_permissions`, `user_roles`, `auth_providers`, `sessions` | Identity, access control, session management |
| **Campaigns** | `campaigns`, `campaign_templates`, `email_templates`, `landing_pages` | Campaign definitions, email content, phishing pages |
| **Targets** | `targets`, `target_groups`, `target_group_members`, `campaign_targets`, `block_list` | Target population management and per-campaign targeting |
| **Infrastructure** | `domain_providers`, `domains`, `dns_records`, `cloud_providers`, `phishing_endpoints`, `smtp_configs`, `campaign_smtp` | External provider credentials, domain management, cloud instances, SMTP |
| **Credential & Tracking** | `captured_credentials`, `tracking_events` | Credential capture and interaction event stream |
| **Logging** | `audit_logs`, `request_logs` | Comprehensive audit trail and endpoint request logging |
| **Reporting** | `report_templates`, `generated_reports` | Report configuration and generated output tracking |
| **AI Readiness** | `ai_providers`, `ai_proposals` | Schema-only in v1; structures for future AI integration |
| **Approval Workflow** | `approval_requests`, `approval_comments` | Approval gates for campaigns and infrastructure operations |

## 4. Table Definitions

### 4.1 Authentication & Authorization Tables

#### 4.1.1 `users`

Stores all user accounts, both locally created and externally provisioned via OIDC, FusionAuth, or LDAP.

**REQ-DB-010** — The `users` table SHALL be defined as follows:

| Column | Type | Nullable | Default | Constraints | Description |
|--------|------|----------|---------|-------------|-------------|
| `id` | `UUID` | NO | `gen_random_uuid()` | `PRIMARY KEY` | Unique user identifier |
| `email` | `TEXT` | NO | | `UNIQUE` | User email address |
| `username` | `TEXT` | NO | | `UNIQUE` | Login username |
| `password_hash` | `TEXT` | YES | | | bcrypt hash; NULL for external auth users |
| `display_name` | `TEXT` | NO | | | Human-readable display name |
| `is_initial_admin` | `BOOLEAN` | NO | `FALSE` | | Marks the first admin created during setup; immutable after creation |
| `auth_provider` | `TEXT` | NO | `'local'` | | Identifier of the auth provider used (matches `auth_providers.type` or `'local'`) |
| `external_id` | `TEXT` | YES | | | User ID from external auth provider; NULL for local accounts |
| `status` | `TEXT` | NO | `'active'` | `CHECK (status IN ('active', 'inactive', 'locked'))` | Account status |
| `created_at` | `TIMESTAMPTZ` | NO | `now()` | | Record creation timestamp |
| `updated_at` | `TIMESTAMPTZ` | NO | `now()` | | Last modification timestamp |

*Acceptance criteria:* The `is_initial_admin` column cannot be changed from `TRUE` to `FALSE` once set (enforced by application logic or trigger). External auth users have `password_hash` as NULL and `external_id` populated.

**Indexes:**
- `idx_users_email` on `(email)` — unique, used for login lookups
- `idx_users_username` on `(username)` — unique, used for login lookups
- `idx_users_auth_provider` on `(auth_provider)` — filters by authentication method
- `idx_users_status` on `(status)` WHERE `status = 'active'` — partial index for active user queries

#### 4.1.2 `roles`

Defines authorization roles. Builtin roles (Admin, Engineer, Operator, Viewer) are seeded at install time and cannot be deleted.

**REQ-DB-011** — The `roles` table SHALL be defined as follows:

| Column | Type | Nullable | Default | Constraints | Description |
|--------|------|----------|---------|-------------|-------------|
| `id` | `UUID` | NO | `gen_random_uuid()` | `PRIMARY KEY` | Unique role identifier |
| `name` | `TEXT` | NO | | `UNIQUE` | Role name (e.g., `admin`, `engineer`, `operator`, `viewer`) |
| `is_builtin` | `BOOLEAN` | NO | `FALSE` | | Builtin roles cannot be deleted or renamed |
| `description` | `TEXT` | YES | | | Human-readable description of the role |
| `created_at` | `TIMESTAMPTZ` | NO | `now()` | | Record creation timestamp |
| `updated_at` | `TIMESTAMPTZ` | NO | `now()` | | Last modification timestamp |

*Acceptance criteria:* Rows with `is_builtin = TRUE` cannot be deleted. The application seeds builtin roles on first migration.

#### 4.1.3 `permissions`

Enumerates individual permission grants. Each permission represents one action on one resource type.

**REQ-DB-012** — The `permissions` table SHALL be defined as follows:

| Column | Type | Nullable | Default | Constraints | Description |
|--------|------|----------|---------|-------------|-------------|
| `id` | `UUID` | NO | `gen_random_uuid()` | `PRIMARY KEY` | Unique permission identifier |
| `resource_type` | `TEXT` | NO | | | Resource being controlled (e.g., `campaign`, `target`, `smtp_config`, `domain`, `user`) |
| `action` | `TEXT` | NO | | | Action on the resource (e.g., `create`, `read`, `update`, `delete`, `approve`, `launch`) |
| `description` | `TEXT` | YES | | | Human-readable description |

**Constraints:**
- `UNIQUE (resource_type, action)` — each resource-action pair is defined only once

*Acceptance criteria:* The combination of `resource_type` and `action` is globally unique. Permissions are seeded by the application during initial migration.

#### 4.1.4 `role_permissions`

*Acceptance criteria:* Deleting a role cascades to remove its permission assignments. Deleting a permission cascades to remove it from all roles.

#### 4.1.6 `auth_providers`

Stores configuration for each enabled authentication provider (local, OIDC, FusionAuth, LDAP).

**REQ-DB-015** — The `auth_providers` table SHALL be defined as follows:

| Column | Type | Nullable | Default | Constraints | Description |
|--------|------|----------|---------|-------------|-------------|
| `id` | `UUID` | NO | `gen_random_uuid()` | `PRIMARY KEY` | Unique provider identifier |
| `type` | `TEXT` | NO | | `CHECK (type IN ('local', 'oidc', 'fusionauth', 'ldap'))` | Provider type |
| `name` | `TEXT` | NO | | `UNIQUE` | Human-readable provider name |
| `configuration` | `BYTEA` | NO | | | Provider-specific configuration (JSONB payload encrypted at rest) |
| `enabled` | `BOOLEAN` | NO | `TRUE` | | Whether this provider is currently active |
| `created_at` | `TIMESTAMPTZ` | NO | `now()` | | Record creation timestamp |
| `updated_at` | `TIMESTAMPTZ` | NO | `now()` | | Last modification timestamp |

**Security:** The `configuration` column stores sensitive data (client secrets, LDAP bind passwords, OIDC client credentials) and SHALL be encrypted at rest using AES-256-GCM before persistence. The application decrypts on read.

*Acceptance criteria:* The `configuration` column stores only encrypted bytes. Reading the raw database value reveals no plaintext secrets. Multiple providers of the same type are allowed (e.g., two OIDC providers) with distinct names.

**Indexes:**
- `idx_auth_providers_type` on `(type)` — lookup by provider type
- `idx_auth_providers_enabled` on `(enabled)` WHERE `enabled = TRUE` — partial index for active providers

#### 4.1.7 `sessions`

Stores active user sessions. Sessions are append-only; expired sessions are purged by a background job.

**REQ-DB-016** — The `sessions` table SHALL be defined as follows:

| Column | Type | Nullable | Default | Constraints | Description |
|--------|------|----------|---------|-------------|-------------|
| `id` | `UUID` | NO | `gen_random_uuid()` | `PRIMARY KEY` | Unique session identifier |
| `user_id` | `UUID` | NO | | `REFERENCES users(id) ON DELETE CASCADE` | Owning user |
| `token_hash` | `TEXT` | NO | | `UNIQUE` | SHA-256 hash of the access token (token itself is never stored) |
| `refresh_token_hash` | `TEXT` | YES | | `UNIQUE` | SHA-256 hash of the refresh token |
| `expires_at` | `TIMESTAMPTZ` | NO | | | Session expiration timestamp |
| `created_at` | `TIMESTAMPTZ` | NO | `now()` | | Session creation timestamp |

**Indexes:**
- `idx_sessions_user_id` on `(user_id)` — lookup sessions by user
- `idx_sessions_token_hash` on `(token_hash)` — unique, token validation lookups
- `idx_sessions_expires_at` on `(expires_at)` — expired session cleanup queries

*Acceptance criteria:* Raw tokens are never stored; only hashes. Deleting a user cascades to delete all their sessions. A background job periodically deletes rows where `expires_at < now()`.

### 4.2 Campaign Tables

#### 4.2.1 `campaigns`

Central table for phishing campaigns. Campaigns progress through a defined lifecycle and are soft-deleted when archived.

**REQ-DB-020** — The `campaigns` table SHALL be defined as follows:

| Column | Type | Nullable | Default | Constraints | Description |
|--------|------|----------|---------|-------------|-------------|
| `id` | `UUID` | NO | `gen_random_uuid()` | `PRIMARY KEY` | Unique campaign identifier |
| `name` | `TEXT` | NO | | | Campaign display name |
| `description` | `TEXT` | YES | | | Free-text campaign description |
| `status` | `TEXT` | NO | `'draft'` | `CHECK (status IN ('draft', 'pending_approval', 'approved', 'building', 'ready', 'active', 'paused', 'completed', 'archived'))` | Campaign lifecycle state |
| `created_by` | `UUID` | NO | | `REFERENCES users(id) ON DELETE RESTRICT` | User who created the campaign |
| `approved_by` | `UUID` | YES | | `REFERENCES users(id) ON DELETE SET NULL` | User who approved the campaign launch |
| `approval_comment` | `TEXT` | YES | | | Comment from the approving user |
| `launched_at` | `TIMESTAMPTZ` | YES | | | Timestamp when the campaign transitioned to `active` |
| `completed_at` | `TIMESTAMPTZ` | YES | | | Timestamp when the campaign completed |
| `archived_at` | `TIMESTAMPTZ` | YES | | | Timestamp when the campaign was archived |
| `configuration` | `JSONB` | YES | `'{}'::jsonb` | | Flexible campaign configuration (send schedules, rate limits, A/B splits, etc.) |
| `deleted_at` | `TIMESTAMPTZ` | YES | | | Soft delete timestamp; NULL when active |
| `created_at` | `TIMESTAMPTZ` | NO | `now()` | | Record creation timestamp |
| `updated_at` | `TIMESTAMPTZ` | NO | `now()` | | Last modification timestamp |

**Indexes:**
- `idx_campaigns_status` on `(status)` WHERE `deleted_at IS NULL` — partial index for active campaign queries
- `idx_campaigns_created_by` on `(created_by)` — lookup campaigns by creator
- `idx_campaigns_launched_at` on `(launched_at)` WHERE `launched_at IS NOT NULL` — time-range queries on launch date
- `idx_campaigns_deleted_at` on `(deleted_at)` WHERE `deleted_at IS NULL` — partial index for non-deleted records

*Acceptance criteria:* Campaign status transitions follow the defined lifecycle. The `created_by` user cannot be deleted while campaigns reference them (ON DELETE RESTRICT). Soft-deleted campaigns remain in the table with `deleted_at` set.

#### 4.2.2 `campaign_templates`

Junction table linking campaigns to email templates, with support for A/B split testing.

**REQ-DB-021** — The `campaign_templates` table SHALL be defined as follows:

| Column | Type | Nullable | Default | Constraints | Description |
|--------|------|----------|---------|-------------|-------------|
| `campaign_id` | `UUID` | NO | | `REFERENCES campaigns(id) ON DELETE CASCADE` | Campaign reference |
| `template_id` | `UUID` | NO | | `REFERENCES email_templates(id) ON DELETE RESTRICT` | Email template reference |
| `split_ratio` | `NUMERIC(5,2)` | NO | `100.00` | `CHECK (split_ratio > 0 AND split_ratio <= 100)` | Percentage of targets receiving this template (must sum to 100 across a campaign) |

**Constraints:**
- `PRIMARY KEY (campaign_id, template_id)` — composite primary key

**Indexes:**
- `idx_campaign_templates_template_id` on `(template_id)` — reverse lookup for template usage

*Acceptance criteria:* For any campaign, the sum of `split_ratio` across all associated templates equals 100. Deleting a template is restricted while it is referenced by a campaign. Deleting a campaign cascades to remove template associations.

#### 4.3.4 `campaign_targets`

Tracks per-target status within a campaign. This is the primary table for campaign progress and metrics.

**REQ-DB-033** — The `campaign_targets` table SHALL be defined as follows:

| Column | Type | Nullable | Default | Constraints | Description |
|--------|------|----------|---------|-------------|-------------|
| `campaign_id` | `UUID` | NO | | `REFERENCES campaigns(id) ON DELETE CASCADE` | Campaign reference |
| `target_id` | `UUID` | NO | | `REFERENCES targets(id) ON DELETE RESTRICT` | Target reference |
| `status` | `TEXT` | NO | `'pending'` | `CHECK (status IN ('pending', 'email_sent', 'email_opened', 'link_clicked', 'credential_submitted', 'reported'))` | Furthest interaction state reached by this target |
| `sent_at` | `TIMESTAMPTZ` | YES | | | Timestamp when the email was sent |
| `opened_at` | `TIMESTAMPTZ` | YES | | | Timestamp of first email open |
| `clicked_at` | `TIMESTAMPTZ` | YES | | | Timestamp of first link click |
| `submitted_at` | `TIMESTAMPTZ` | YES | | | Timestamp of credential submission |

**Constraints:**
- `PRIMARY KEY (campaign_id, target_id)` — composite primary key; each target appears at most once per campaign

**Indexes:**
- `idx_campaign_targets_target_id` on `(target_id)` — reverse lookup for target history
- `idx_campaign_targets_status` on `(campaign_id, status)` — campaign progress aggregation
- `idx_campaign_targets_sent_at` on `(campaign_id, sent_at)` WHERE `sent_at IS NOT NULL` — delivery timeline queries

*Acceptance criteria:* Status progresses monotonically (e.g., a target at `link_clicked` does not regress to `email_sent`). Targets cannot be deleted while they have campaign associations (ON DELETE RESTRICT).

### 4.4 Infrastructure Tables

**REQ-DB-100** — All foreign key columns SHALL have corresponding indexes to prevent table-scan performance degradation on JOIN operations and cascading deletes.

*Acceptance criteria:* Every `REFERENCES` column has an explicit index definition (either as part of a primary key or as a standalone index).

**REQ-DB-101** — Composite indexes SHALL be defined for the following high-frequency query patterns:

| Index | Table | Columns | Purpose |
|-------|-------|---------|---------|
| `idx_campaign_targets_status` | `campaign_targets` | `(campaign_id, status)` | Campaign progress dashboards and metric aggregation |
| `idx_tracking_events_campaign_event_type` | `tracking_events` | `(campaign_id, event_type, created_at)` | Metric queries: count events by type over time for a campaign |
| `idx_audit_logs_category` | `audit_logs` | `(category, timestamp)` | Filtered log browsing by category with time ordering |

| `idx_tracking_events_campaign_target` | `tracking_events` | `(campaign_id, target_id)` | Per-target event timeline within a campaign |

*Acceptance criteria:* Each listed composite index exists in the migration. Query EXPLAIN plans for the corresponding query patterns show index usage (no sequential scans on large tables).

**REQ-DB-102** — Partial indexes SHALL be used on status columns to optimize queries that filter for active records:

| Index | Table | Condition | Purpose |
|-------|-------|-----------|---------|
| `idx_campaigns_status` | `campaigns` | `WHERE deleted_at IS NULL` | Active campaign queries |
| `idx_targets_deleted_at` | `targets` | `WHERE deleted_at IS NULL` | Active target queries |
| `idx_phishing_endpoints_status` | `phishing_endpoints` | `WHERE status NOT IN ('terminated')` | Active endpoint queries |

| `idx_users_status` | `users` | `WHERE status = 'active'` | Active user queries |

*Acceptance criteria:* Partial indexes are defined with the correct WHERE clause. Queries that match the partial index condition use the index (verified via EXPLAIN).

**REQ-DB-103** — A GIN index SHALL be created on `audit_logs.details` to support full-text search across structured log data.

*Acceptance criteria:* The GIN index exists. JSONB containment queries (`@>`) and key-existence queries (`?`) against `audit_logs.details` use the index.

**REQ-DB-104** — A GIN index SHALL be created on `generated_reports.campaign_ids` to support array containment queries (finding all reports that include a given campaign).

*Acceptance criteria:* Queries using `campaign_ids @> ARRAY[<uuid>]::uuid[]` use the GIN index.

## 6. Encryption Requirements

**REQ-DB-110** — The following columns SHALL be encrypted at rest using application-level AES-256-GCM encryption:

| Table | Column | Content |
|-------|--------|---------|
| `auth_providers` | `configuration` | OIDC client secrets, LDAP bind passwords, FusionAuth API keys |
| `domain_providers` | `credentials` | Namecheap API keys, GoDaddy API keys, AWS access keys, Azure client secrets |
| `cloud_providers` | `credentials` | AWS access key ID + secret, Azure tenant/client/secret, IAM role ARNs |
| `smtp_configs` | `password` | SMTP authentication passwords and OAuth tokens |
| `captured_credentials` | `form_data` | Credentials submitted by phishing targets |
| `ai_providers` | `configuration` | AI provider API keys and endpoint configuration |

*Acceptance criteria:* Each encrypted column stores ciphertext (BYTEA or base64 TEXT). A database dump of these columns reveals no plaintext secrets. The application encryption key is sourced from the `TACKLE_ENCRYPTION_KEY` environment variable. Rotating the encryption key requires a documented re-encryption procedure.

**REQ-DB-111** — Encrypted columns SHALL include an authenticated encryption nonce/IV as part of the stored ciphertext. Each encryption operation SHALL use a unique nonce.

*Acceptance criteria:* No two encrypted values in the database share the same nonce. The encryption implementation follows NIST guidelines for AES-GCM nonce handling.

**REQ-DB-112** — The application SHALL provide a key rotation utility that re-encrypts all encrypted columns with a new encryption key without downtime.

*Acceptance criteria:* A CLI command or API endpoint can re-encrypt all secrets from old key to new key. The operation is idempotent and can be resumed if interrupted.

## 7. Migration Strategy

**REQ-DB-120** — Database schema changes SHALL be managed through sequential, numbered migration files using a Go-based migration tool (e.g., golang-migrate with local import path).

*Acceptance criteria:* All schema DDL is defined in migration files, never applied ad-hoc. The migration tool is imported via a local Go module path (no `github.com` in import paths).

**REQ-DB-121** — Each migration SHALL consist of an up migration (applying the change) and a down migration (reverting the change). Both files SHALL be present for every migration.

*Acceptance criteria:* Every migration directory entry contains both `NNNNNN_description.up.sql` and `NNNNNN_description.down.sql` files. Down migrations are tested to verify they cleanly revert the corresponding up migration.

**REQ-DB-122** — The migration tool SHALL track schema version state in a dedicated database table (e.g., `schema_migrations`). The application SHALL refuse to start if the database schema version does not match the expected version.

*Acceptance criteria:* A `schema_migrations` table (or equivalent) exists. Running the application against an outdated schema produces a clear error message instructing the operator to run migrations.

**REQ-DB-123** — Migrations SHALL be backward-compatible where possible. Destructive changes (dropping columns, changing types) SHALL be performed in multi-step migrations with a deprecation period.

*Acceptance criteria:* No migration drops a column or table that is still referenced by application code. Destructive migrations are documented with a migration guide.

**REQ-DB-124** — The initial migration SHALL create all tables, indexes, constraints, and seed data (builtin roles, default permissions) defined in this document.

*Acceptance criteria:* Running the initial migration against an empty database produces a fully functional schema matching all table definitions in this document. Builtin roles (`admin`, `engineer`, `operator`, `viewer`) and their default permissions are seeded.

## 8. Security Considerations

**SEC-DB-001** — The PostgreSQL database SHALL be configured to accept connections only from the framework server application. Network-level access controls (firewall rules, security groups) SHALL restrict database port access.

**SEC-DB-002** — The database connection SHALL use TLS encryption in transit. The application SHALL verify the database server's TLS certificate.

**SEC-DB-003** — The database user account used by the application SHALL have the minimum privileges required: SELECT, INSERT, UPDATE, DELETE on application tables, and the ability to run migrations. The database user SHALL NOT have superuser or createdb privileges in production.

**SEC-DB-004** — Application-level encryption keys SHALL never be stored in the database. The key SHALL be provided exclusively via the `TACKLE_ENCRYPTION_KEY` environment variable.

**SEC-DB-005** — Database backups SHALL be encrypted. The backup encryption key SHALL be distinct from the application-level field encryption key.

**SEC-DB-006** — All SQL queries SHALL use parameterized queries (prepared statements). No string concatenation or interpolation SHALL be used to construct SQL. The Go application SHALL use a query builder or ORM that enforces parameterization.

**SEC-DB-007** — The `captured_credentials` table SHALL be subject to additional access controls. Only users with the explicit `captured_credentials:read` permission SHALL be able to decrypt and view form data. Access to this table SHALL be logged in the audit trail.

## 9. Performance Considerations

**REQ-DB-130** — The `tracking_events` and `request_logs` tables are expected to be the highest-volume tables. The schema SHALL support table partitioning (by `created_at` range) as a future optimization without requiring application code changes.

*Acceptance criteria:* The table definitions are compatible with PostgreSQL declarative partitioning. The application does not use features that prevent partitioning (e.g., cross-partition foreign keys on the partitioned table's non-key columns).

**REQ-DB-131** — The `audit_logs` table SHALL support retention policies. A background job SHALL be able to archive or delete log entries older than a configurable threshold (default: 365 days) without impacting active queries.

*Acceptance criteria:* A configurable retention period exists. The cleanup job deletes or archives in batches to avoid long-running transactions and lock contention.

**REQ-DB-132** — Database connection pooling SHALL be configured at the application level. The default pool size SHALL be configurable via environment variable (default: 25 connections).

*Acceptance criteria:* The application uses a connection pool (e.g., pgxpool). The pool size is configurable. Connection exhaustion produces a clear error rather than hanging.

## 10. Acceptance Criteria (Schema-Wide)

- [ ] All 33 tables defined in this document exist in the initial migration
- [ ] Every table uses UUID primary keys with `gen_random_uuid()` default; no SERIAL or BIGSERIAL columns exist
- [ ] Every table has `created_at` (and `updated_at` where applicable) with `TIMESTAMPTZ` type and `NOT NULL DEFAULT now()`
- [ ] All foreign key relationships have explicit ON DELETE behavior (CASCADE, SET NULL, or RESTRICT)
- [ ] Every foreign key column has a corresponding index
- [ ] Soft-delete tables (`campaigns`, `targets`, `target_groups`, `domains`, `email_templates`, `landing_pages`, `smtp_configs`) have a `deleted_at TIMESTAMPTZ` column
- [ ] All status/type columns use TEXT with CHECK constraints (no native ENUM types)
- [ ] All encrypted columns (`auth_providers.configuration`, `domain_providers.credentials`, `cloud_providers.credentials`, `smtp_configs.password`, `captured_credentials.form_data`, `ai_providers.configuration`) store only ciphertext
- [ ] The GIN index on `audit_logs.details` supports JSONB full-text search
- [ ] The GIN index on `generated_reports.campaign_ids` supports array containment queries
- [ ] Partial indexes exist for all specified active-record query patterns
- [ ] Composite indexes exist for all specified high-frequency query patterns
- [ ] Up and down migration files exist for every schema change
- [ ] The `schema_migrations` table tracks the current schema version
- [ ] Builtin roles and default permissions are seeded by the initial migration
- [ ] Running the full migration suite against an empty database succeeds without errors
- [ ] Running all down migrations in reverse order returns the database to an empty state
