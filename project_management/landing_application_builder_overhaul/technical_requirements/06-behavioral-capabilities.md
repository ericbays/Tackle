# 06 — Behavioral Capabilities

## 6.1 Overview

Behavioral capabilities are data collection mechanisms that go beyond explicit form submission. They allow the landing application to passively collect information about the target's behavior, environment, and session — all without requiring the target to click "Submit." These capabilities are configured by the operator in the builder at either the page level (broad capabilities) or the component level (targeted capabilities).

All behavioral data is collected by JavaScript running in the target's browser, sent to the landing application's Go backend, and forwarded upstream to Tackle's metrics/capture endpoints. The JavaScript is generated by the `servergen` pipeline based on the operator's builder configuration — the operator never writes code.

## 6.2 Configuration Levels

### Page-Level Behaviors

Broad capabilities that apply to the entire page. Configured in the Page Manager's behavior settings.

| Capability | Description |
|------------|-------------|
| Browser Fingerprinting | Collect hardware, browser, and screen information |
| Session Token Extraction | Scrape localStorage, sessionStorage, and cookies |
| Page View Tracking | Report page load events with timing data |
| Scroll Depth Tracking | Report how far the target scrolls on the page |
| Time-on-Page Tracking | Report how long the target spends on the page |

### Component-Level Behaviors

Targeted capabilities attached to specific components. Configured in the component's Advanced tab.

| Capability | Applicable To | Description |
|------------|--------------|-------------|
| Keystroke Capture | Text inputs, password inputs, email inputs, textareas | Capture individual keystrokes as they're typed |
| Clipboard Capture | Text inputs, textareas | Capture content pasted from clipboard |
| Focus/Blur Tracking | Any input component | Report when a field gains or loses focus |
| Click Tracking | Any component | Report click events on specific elements |

## 6.3 Page-Level Capabilities

### 6.3.1 Browser Fingerprinting

Collects a comprehensive fingerprint of the target's browser and hardware environment on page load.

**Data collected**:

| Category | Fields |
|----------|--------|
| **Screen** | Width, height, color depth, pixel ratio, available dimensions |
| **Browser** | User agent, platform, language, languages, cookie enabled, do-not-track |
| **Hardware** | Hardware concurrency (CPU cores), device memory, max touch points |
| **Graphics** | WebGL renderer, WebGL vendor (via canvas fingerprinting) |
| **Timezone** | Timezone name, UTC offset |
| **Plugins** | Installed browser plugins list |
| **Features** | WebRTC support, WebGL support, canvas support, audio context support |

**Trigger**: Executes once on page load. Data is sent to the landing app's metrics endpoint as a `browser_fingerprint` event.

**Builder configuration**:
```
Page Behaviors:
  [✓] Browser Fingerprinting
```

### 6.3.2 Session Token Extraction

Scrapes stored authentication tokens and session data from the target's browser.

**Targets**:

| Storage | What's Extracted |
|---------|-----------------|
| **Cookies** | All cookies accessible to the page domain |
| **localStorage** | All key-value pairs |
| **sessionStorage** | All key-value pairs |

**Trigger**: Executes once on page load (after a brief delay to allow any client-side auth flows to complete). Data is sent as a `session_capture` event.

**Builder configuration**:
```
Page Behaviors:
  [✓] Session Token Extraction
      Delay before capture: [500] ms
      Scope:
        [✓] Cookies
        [✓] localStorage
        [✓] sessionStorage
```

### 6.3.3 Page View Tracking

Reports when the target loads and views a page.

**Data collected**:
- Page route
- Timestamp
- Referrer
- Page load timing (navigation start to DOM content loaded)

**Trigger**: On page load (DOMContentLoaded). Sent as a `page_view` event.

**Builder configuration**:
```
Page Behaviors:
  [✓] Page View Tracking
```

### 6.3.4 Scroll Depth Tracking

Reports how far the target scrolls down the page.

**Data collected**:
- Maximum scroll depth (percentage of page height)
- Scroll depth checkpoints (25%, 50%, 75%, 100%)

**Trigger**: Debounced scroll event listener. Reports on scroll stop (200ms debounce) and on page unload. Sent as `scroll_depth` events.

**Builder configuration**:
```
Page Behaviors:
  [✓] Scroll Depth Tracking
```

### 6.3.5 Time-on-Page Tracking

Reports how long the target spends on the page.

**Data collected**:
- Total time on page (ms)
- Active time (ms) — excludes time when the tab is not focused
- Idle periods

**Trigger**: Reports on page unload (beforeunload event) and periodically (every 30 seconds while active). Sent as `time_on_page` events.

**Builder configuration**:
```
Page Behaviors:
  [✓] Time-on-Page Tracking
      Report interval: [30] seconds
```

## 6.4 Component-Level Capabilities

### 6.4.1 Keystroke Capture

Captures individual keystrokes as the target types into a specific input field. This operates independently of form submission — data is captured in real-time, keystroke by keystroke.

**Data collected per keystroke**:
- Key pressed
- Timestamp
- Field name (HTML name attribute)
- Cumulative field value at time of keystroke

**Trigger**: `keydown` event listener on the targeted input element. Keystrokes are buffered and sent in batches (every 2 seconds or on field blur). Sent as `keystroke_capture` events.

**Builder configuration** (in input component's Advanced tab):
```
Component Behaviors:
  [✓] Keystroke Capture
      Buffer interval: [2000] ms
```

**Applicable component types**: text_input, password_input, email_input, textarea

### 6.4.2 Clipboard Capture

Captures content that the target pastes into an input field.

**Data collected**:
- Pasted text content
- Field name
- Timestamp

**Trigger**: `paste` event listener on the targeted input element. Sent immediately as a `clipboard_capture` event.

**Builder configuration** (in input component's Advanced tab):
```
Component Behaviors:
  [✓] Clipboard Capture
```

**Applicable component types**: text_input, password_input, email_input, textarea

### 6.4.3 Focus/Blur Tracking

Reports when a target interacts with specific form fields — when they click into a field (focus) and when they leave it (blur).

**Data collected**:
- Event type (focus or blur)
- Field name
- Field value at time of blur
- Timestamp
- Duration of focus (on blur events)

**Trigger**: `focus` and `blur` event listeners. Sent as `field_interaction` events.

**Builder configuration** (in input component's Advanced tab):
```
Component Behaviors:
  [✓] Focus/Blur Tracking
```

**Applicable component types**: All input components (text_input, password_input, email_input, textarea, select, checkbox, radio)

### 6.4.4 Click Tracking

Reports when a target clicks on a specific element. Useful for tracking button clicks, link clicks, and interactions with non-form elements.

**Data collected**:
- Element DOM ID
- Element type
- Click coordinates (relative to element)
- Timestamp

**Trigger**: `click` event listener on the targeted element. Sent as `element_click` events.

**Builder configuration** (in any component's Advanced tab):
```
Component Behaviors:
  [✓] Click Tracking
```

**Applicable component types**: All component types

## 6.5 Data Flow

All behavioral data follows the same pipeline:

```
Browser JavaScript (generated by servergen)
        │
        │  Async POST (XMLHttpRequest or fetch)
        │  to landing app's internal metrics endpoint
        ▼
Landing App Go Backend
        │
        │  POST to Tackle's /api/v1/internal/metrics
        │  (or /api/v1/internal/captures for keystroke/clipboard data)
        ▼
Tackle Framework
        │
        │  Stores in metrics/captures tables
        │  Updates real-time dashboard
        ▼
Operator Dashboard
```

### Internal Metrics Endpoint

The `servergen` pipeline generates a Go handler at a randomized internal path (not the operator-defined form action paths) that receives behavioral data from the page's JavaScript. This endpoint:

1. Accepts JSON payloads with event type, data, and metadata
2. Forwards the payload to Tackle's internal metrics API
3. Returns a minimal 200 response

The path to this endpoint is embedded in the generated JavaScript and is not visible to the target in normal browsing. It is styled as a plausible API call (e.g., `/api/v2/telemetry`, `/metrics/collect`) to avoid suspicion if a defender inspects network traffic.

## 6.6 Stealth Considerations

### JavaScript Injection

The behavioral JavaScript is:

- Bundled with the React application code during `servergen` compilation
- Not loaded from an external source
- Not identifiable as a separate "tracking script" in the page source
- Embedded inline within the application's main JS bundle

### Network Requests

Behavioral data is sent via:

- Standard `fetch` or `XMLHttpRequest` calls to the landing app's own domain
- POST requests to internal endpoints that look like normal API calls
- No cross-origin requests
- No requests to known tracking domains

### Timing and Throttling

- Behavioral data is batched and debounced to avoid generating suspicious volumes of network traffic
- Keystroke data is buffered (default 2 seconds)
- Scroll depth is debounced (200ms)
- Time-on-page reports at configurable intervals (default 30 seconds)

## 6.8 OAuth Consent Simulation

A specialized behavioral capability for simulating OAuth/SSO consent flows.

### Purpose

Allows the operator to build a page that mimics an OAuth consent screen (e.g., "Contoso wants to access your account"). When the target "grants" consent, the landing application captures the interaction and can be configured to redirect to a real OAuth endpoint.

### Configuration

This is not a standalone behavior toggle but rather a combination of:

1. **Visual**: A page designed to look like an OAuth consent screen (using the builder's standard components — heading, paragraph, logo, buttons, checkboxes for permissions)
2. **Capture**: A form wrapping the "consent" elements, with the submit button labeled "Allow" or "Grant Access"
3. **Post-capture**: Configured to redirect to the real service or show a success message
4. **Tracking**: Click tracking on the consent button, page view tracking on the consent page

The builder provides no special "OAuth" component — the operator builds the consent UI using standard components and captures the interaction using standard form capture.

## 6.9 Capability Matrix

| Capability | Page-Level | Component-Level | Data Destination | Stealth Risk |
|-----------|-----------|----------------|-----------------|-------------|
| Page View Tracking | Yes | — | metrics | Very Low |
| Time-on-Page | Yes | — | metrics | Very Low |
| Scroll Depth | Yes | — | metrics | Low |
| Browser Fingerprinting | Yes | — | metrics | Low |
| Session Token Extraction | Yes | — | captures | Medium |
| Keystroke Capture | — | Yes (inputs) | captures | Medium |
| Clipboard Capture | — | Yes (inputs) | captures | Low |
| Focus/Blur Tracking | — | Yes (inputs) | metrics | Very Low |
| Click Tracking | — | Yes (any) | metrics | Very Low |

**Stealth Risk** indicates how likely the behavior is to be detected by defensive tooling or suspicious targets. "Very Low" means the behavior generates no suspicious network activity. "Medium" means the behavior generates additional network requests that could be noticed in browser dev tools.
